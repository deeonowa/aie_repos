## 1. Паспорт проекта

**Название проекта:** CVE Severity Triage API — сервис первичной приоритизации уязвимостей по тексту описания CVE.

**Автор:** `Зуенко Андрей Андреевич`

**Группа:** `Группа БИСО-01-23`

**Контакт:** `@deeonowa`

**Ссылка на репозиторий:** `https://github.com/deeonowa/aie_repos/tree/main/project`

**Краткое описание.**

В проекте реализован сквозной ML-сервис для аналитика vulnerability management: по английскому тексту описания CVE из NVD модель относит уязвимость к классу **low / medium / high** (пороги CVSS v3 base 4.0 и 7.0). Использованы открытые данные NVD за **2024 и 2025** (~78 040 записей), классические модели на TF-IDF (Dummy, Logistic Regression, SGD, Random Forest) и REST API на FastAPI. Итог работы — воспроизводимый пайплайн (загрузка данных → обучение → артефакт модели → сервис с эндпоинтами `/health` и `/predict`), ноутбуки с EDA и экспериментами, автотесты и документированная структура.

## 2. Постановка задачи и контекст

### 2.1. Предметная область и задача

**Тип задачи:** многоклассовая **классификация** текста (3 класса).

**Предметная область:** информационная безопасность, управление уязвимостями. В National Vulnerability Database (NVD) публикуются тысячи CVE с текстовым описанием и числовой оценкой CVSS. Аналитику необходимо быстро отделять записи с высокой общей технической серьёзностью от менее критичных.

**Потенциальный пользователь:** аналитик vulnerability management или SOC.

**Сценарий использования:** аналитик копирует или получает из NVD текст описания CVE, отправляет его в HTTP API и получает метку `low` / `medium` / `high` и вероятности по всем классам. Результат применяется для **первичной сортировки очереди** разбора, до детальной оценки риска для конкретной инфраструктуры.

### 2.2. Формулировка задачи в терминах ML/ИИ

| Элемент | Содержание |
|---------|------------|
| **Входные данные** | Одно текстовое поле — описание уязвимости на английском языке (как в NVD) |
| **Целевая переменная** | `severity` (low, medium, high) (`severity_id` 0 / 1 / 2) |
| **Правило разметки** | low: CVSS score < 4; medium: 4.0 ≤ CVSS score < 7.0; high: CVSS score ≥ 7.0 |
| **Выход модели** | Вероятности трёх классов и argmax - `risk` |
| **Требования** | Высокий recall по классу high; сбалансированный macro F1; компактный артефакт для API |

Метка отражает **официальную серьёзность NVD**, а не риск для конкретной организации: в проекте нет данных об активах, факте эксплуатации и компенсирующих мерах.

### 2.3. Целевые метрики качества

| Метрика | Назначение |
|---------|------------|
| **ROC-AUC (OvR weighted)** | Качество ранжирования в многоклассовой постановке one-vs-rest |
| **Macro F1** | Сводная метрика по всем трём классам при дисбалансе |
| **Recall (класс high)** | Доля высоких CVE, не отнесённых к lower-классам; **основная** прикладная метрика |
| **Precision (класс high)** | Доля верных срабатываний среди предсказанных high |
| **F1 (класс high)** | Сводный баланс precision и recall для критичного класса |
| **Accuracy** | Справочная метрика на фоне умеренного дисбаланса классов |

**Recall по классу high** выбран главным критерием: при triage пропуск потенциально критичной уязвимости важнее, чем лишнее срабатывание, которое аналитик отфильтрует при ручной проверке.

## 4. Модели и подходы

### 4.1. Базовые (baseline) модели

В качестве baseline обучен **DummyClassifier** (`strategy='most_frequent'`) в общем `Pipeline` с **TfidfVectorizer** (те же признаки, что у остальных моделей).

**Результаты на test:** macro F1 0.22, accuracy 0.50 — нижняя граница; модель предсказывает только мажоритарный класс (high), recall_high = 1.0 при нулевом recall по low/medium.

### 4.2. Улучшенные модели и эксперименты

Поверх baseline обучены три модели на **общих TF-IDF-признаках**:

| Параметр TF-IDF | Значение |
|-----------------|----------|
| `max_features` | 30 000 |
| `ngram_range` | (1, 2) |
| `min_df` | 2 |
| `stop_words` | english |

| Модель | Ключевые настройки |
|--------|-------------------|
| **LogisticRegression** | `class_weight='balanced'`, `max_iter=1000`, `random_state=42` |
| **SGDClassifier** | `loss='log_loss'`, `class_weight='balanced'` |
| **RandomForestClassifier** | `n_estimators=200`, `class_weight='balanced'`, `random_state=42` |

Подбор гиперпараметров выполнялся отдельно, процесс не зафиксирован в проекте; сравнивались семейства моделей при фиксированном векторизаторе. Параметры зафиксированы в `configs/config.yaml`.

**Эксперименты** — **`notebooks/02_experiments.ipynb`** (сравнение на подвыборке, `metrics_all.json`, графики).  
**Финальное обучение** — **`python -m src.train`** (`src/models/train.py`, полный датасет, `joblib`).

### 4.3. Нейросетевые модели

Нейросетевые архитектуры **не применялись**. На объёме ~78k записей TF-IDF с линейными и ансамблевыми моделями дали приемлемое качество при умеренной сложности пайплайна.

### 5.1. Экспериментальный протокол

| Параметр | Значение |
|----------|----------|
| Разбиение | Train / test = **80% / 20%**

| Обучение | `fit` на train (**62 432** записей) |
| Метрики | На **test** (**15 608** записей) |

### 5.2. Сравнение моделей по метрикам

В `artifacts/metrics_all.json` (подвыборка) и `artifacts/best_model.json` (финальная модель) сохраняются метрики:
`accuracy`, `f1` (macro), `f1_weighted`, `balanced_accuracy`, `mcc`, `roc_auc`.

**Короткий вывод по подвыборке:** LogisticRegression показывает лучший баланс **macro F1** и **balanced_accuracy**; RandomForestClassifier чаще лидирует по `accuracy`/`mcc`, но слабее по macro F1.

Baseline всегда предсказывает класс high (recall_high = 1.0) при нулевом качестве по остальным классам.

Метрики всех моделей (подвыборка): `artifacts/metrics_all.json`. Финальная модель: `artifacts/best_model.json`.

### 5.3. Выбор финальной модели

**Финальная модель для API:** **Logistic Regression**.

**Обоснование:**

- по подвыборке LogReg даёт лучший баланс по **macro F1** и **balanced_accuracy**;
- RF даёт более высокий `accuracy`/`mcc`, но в среднем уступает LogReg по macro F1 и тяжелее для production;
- артефакт LogReg компактнее, инференс быстрее, сопровождение проще;
- GDClassifier не выбран, так как на подвыборке macro F1 и balanced_accuracy ниже, чем у Logistic Regression, при сопоставимых accuracy и ROC-AUC.

**Trade-off:** выбор LogReg — компромисс между качеством, скоростю и размером для API.

## 6. Архитектура решения и сервис

### 6.1. Архитектура пайплайна

**Офлайн (подготовка):** скрипт `python -m src.data` скачивает feed NVD за 2024–2025, извлекает английское описание и CVSS v3, строит метки `low` / `medium` / `high` (пороги 4.0 и 7.0) и сохраняет `data/processed/cve_dataset.csv`. В ноутбуках выполняются EDA и сравнение моделей; команда `python -m src.train` обучает финальный пайплайн **TF-IDF + Logistic Regression** и сохраняет `artifacts/cve_model.joblib`.

**Онлайн (сервис):** FastAPI при старте загружает joblib-артефакт в память. На запрос `/predict` текст описания CVE проходит `clean_description` (нормализация пробелов, обрезка длины), затем sklearn-pipeline возвращает класс серьёзности и вероятности по трём классам.

```
NVD (2024+2025) → load_nvd → cve_dataset.csv → train → cve_model.joblib
                                                          ↓
                                POST /predict (текст) → api → predict_text → JSON
```

### 6.2. API и endpoints

Реализован сервис на **FastAPI**.

| Endpoint | Метод | Назначение |
|----------|-------|------------|
| `/health` | GET | Статус сервиса, загрузка модели, пороги CVSS |
| `/predict` | POST | Предсказание класса серьёзности по тексту CVE |
| `/docs` | GET | Swagger UI |

**Запрос `/predict`:**

```json
{
  "description": "текст описания CVE, 10–8000 символов"
}
```

Ответ /predict:
```json
{
  "risk": "high",
  "probabilities": {"low": 0.0012, "medium": 0.024, "high": 0.9748},
  "probability": 0.9748,
  "model_name": "LogisticRegression",
  "latency_ms": 4.5,
  "model_version": "2.0.0"
}
```

### 6.3. Технологический стек и запуск

| Компонент | Технология |

```bash
cd project
```

```bash

python -m venv .venv
source .venv/bin/activate   # Windows: .venv\Scripts\activate
pip install -r requirements.txt

```

Сервис доступен на http://localhost:8000.

Обучение (python -m src.train) и загрузка данных NVD (python -m src.data) выполняются на хосте. Docker-образ содержит только inference API и готовую модель artifacts/cve_model.joblib.

## 7. Наблюдаемость, конфигурация и безопасность

### 7.1. Логи и наблюдаемость

В `src/service/api.py` настроено логирование уровня **INFO**.

При вызове `/predict` в лог записываются: `risk`, `probability`, длина входного текста, `latency_ms`. Полный текст CVE **не логируется**.

Корректность работы проверяется запросом **`GET /health`**: при загруженной модели возвращаются `"status": "ok"`, `"model_loaded": true`, `"model_name": "logreg"`.

### 7.2. Конфигурации

| Файл | Содержание |
|------|------------|
| `configs/config.yaml` | Годы NVD, пороги 4.0/7.0, TF-IDF, `production_model`, порт сервиса |
| `configs/.env.example` | Шаблон переменных: `LOG_LEVEL`, `SERVICE_HOST`, `SERVICE_PORT`, `MODEL_PATH` |

Переменные окружения переопределяют порт, путь к модели и уровень логов без правки кода.

### 7.3. Безопасность

- API-ключи и пароли **не используются**; открытый feed NVD не требует аутентификации.
- Файл `.env` в **`.gitignore`**; в репозитории только `configs/.env.example` без секретов.
- В репозитории — только открытые тексты CVE; персональные и служебные данные отсутствуют.
- Вход API валидируется через Pydantic (длина текста 10–8000 символов).
